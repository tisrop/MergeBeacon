# 桌面通知能力边界与系统验收

本文记录 MergeBeacon 桌面通知的运行边界、限流观测方式、系统级验收步骤，以及后台常驻和
服务端推送方案的评估结论。自动化测试只能验证应用逻辑，不能替代操作系统通知中心实测。

## 当前实现

- Vue `NotificationManager` 在应用进程内每 10 分钟轮询一次已启用且已登录的平台。
- 每个平台分别请求“评审请求”和“我创建的”两个收件箱类别。
- 单个类别成功时保留该类别结果，不因另一个类别限流而暂停整个平台。
- 两个类别都失败且包含限流错误时，当前平台退避 15 分钟，其他平台继续轮询。
- 设置页显示每个平台的最近检查、最近成功、失败类别、本次运行限流次数和重试倒计时。
- 首次成功轮询只建立基线，不为已有条目补发通知。
- 通知权限、网络、轮询、发送和点击监听错误均显示在应用内。

限流识别当前依赖平台错误字符串中的 `429`、`rate limit` 或同义文本。后端错误仍通过字符串
返回，尚未提供结构化的 HTTP 状态、限流配额或平台 `Retry-After`。

## 生命周期边界

桌面通知轮询依赖主应用进程和 WebView 中的 JavaScript timer：

- 主应用运行时，窗口处于前台或后台都可以继续轮询，但操作系统仍可能因睡眠、节能或网络状态
  延后 timer。
- 退出 MergeBeacon 后，进程和 timer 都会结束，不再请求 GitHub、GitLab 或 Gitee。
- 当前没有系统托盘常驻进程、登录时启动项、LaunchAgent、Windows Service、systemd user service
  或独立后台 worker。
- 当前没有 MergeBeacon 服务端，也没有接收平台 Webhook 后转发 APNs、WNS 或其他推送的能力。
- 重新启动应用后会使用本地快照继续比较；超过 30 天的通知快照会被清理。

因此，“启用桌面通知”只表示应用运行期间启用低频检查，不表示退出应用后仍能收到通知。

## 系统级验收矩阵

| 系统 | 通知后端 | 测试版本 | 状态 | 备注 |
|---|---|---|---|---|
| macOS | Notification Center | macOS 15.7.7 | 核心路径通过 | Debug `.app` bundle，结果见下文 |
| Windows | Windows toast | Windows 10/11 | 未执行 | 需要 Windows 安装包和真实 WebView2 环境 |
| Linux | D-Bus 通知服务 | WebKitGTK 支持范围 | 未执行 | 需要分别验证 GNOME/KDE 与通知 daemon |

每个平台必须使用安装后的 bundle 验证以下项目：

1. 首次启用时出现系统权限请求；拒绝后应用保持关闭状态。
2. 点击“发送测试通知”后，系统通知中心显示标题和正文。
3. 测试通知不显示“打开 Pull Request”动作。
4. 私有仓库通知不显示仓库名、PR / MR 标题或代码信息。
5. 普通通知点击后打开正确的平台、仓库和 PR / MR。
6. 在系统设置撤销权限后，下一次检查停止轮询并显示应用内错误。
7. 完整 `429` 显示平台、重试倒计时和失败类别；其他平台继续轮询。
8. 仅一个类别 `429` 时显示部分成功，但不启用平台级退避。
9. 系统睡眠、唤醒和网络恢复后不会并发启动重复轮询。
10. 完全退出应用后，等待超过 10 分钟，确认没有新的平台请求和通知。

Linux 还必须记录桌面环境、WebKitGTK 版本、通知 daemon 和沙箱/打包方式；单一发行版通过不能
代表所有 Linux 桌面环境通过。

### macOS 15.7.7 实测结果

2026-07-20 使用 `MergeBeacon.app` Debug bundle 验证：

- 第一次发送测试通知时出现 macOS 原生授权对话框。
- 允许后重新发送，Notification Center 显示“MergeBeacon 测试通知”和完整生命周期边界正文。
- 测试通知只提供“显示详细信息”和“关闭”，没有“打开 Pull Request”动作。
- 设置页显示“测试通知已交给系统通知服务”，退出 bundle 后进程停止。

本次没有撤销现有系统权限，也没有构造包含真实仓库信息的远端事件，因此权限撤销、私有内容和
真实 PR / MR 点击定位仍需要在隔离测试账号中验收。Windows 和 Linux 未在当前 macOS 主机执行，
不能根据 Tauri 插件或自动化测试推断为通过。

## 后台常驻与服务端推送评估

| 方案 | 退出主窗口后通知 | 完全退出后通知 | 主要成本 | 结论 |
|---|---:|---:|---|---|
| 当前 WebView 轮询 | 取决于应用是否仍运行 | 不支持 | API 配额、timer 延迟 | 当前版本保留 |
| Tauri 托盘常驻 + Rust 轮询 | 支持 | 不支持 | 托盘/自启动、跨系统生命周期、Rust 调度与电源管理 | 可作为下一阶段 |
| 操作系统计划任务或服务 | 支持 | 部分支持 | 凭证访问、安装权限、卸载清理、三套系统实现 | 不建议 |
| MergeBeacon 服务端 + 平台 Webhook/推送 | 支持 | 支持 | 服务端、账号绑定、Webhook、推送证书、隐私与运维 | 独立产品决策 |

### 推荐路径

短期保持当前应用内低频轮询，并完善状态观测和系统验收。若需要关闭主窗口后继续通知，优先评估
“关闭到托盘”模式，并将轮询从 WebView timer 移到 Rust 后台任务；这仍不能覆盖用户明确退出或
系统终止进程的情况。

只有“应用完全退出后仍通知”成为明确产品要求时，才进入服务端推送设计。该方案需要先确定：

- GitHub App、GitLab OAuth/Application 和 Gitee 授权模型；
- SaaS 与自托管平台的 Webhook 可达性；
- 设备注册、推送证书、撤销和多设备去重；
- 私有仓库元数据的最小化存储、加密、保留和删除策略；
- Webhook 重放、防伪、限流、顺序和离线补偿；
- 服务端成本、可用性目标和用户关闭远端通知的方式。

在这些条件确定前，不应在客户端文案中承诺应用退出后的通知能力。
