# AI 评审规则与历史同步方案

## 当前能力

MergeBeacon 当前不提供团队服务端或跨设备同步服务。团队共享规则应提交到仓库，并随代码版本一同
审查。客户端会在当前 PR/MR 的 `head_sha` 上按以下顺序尝试读取第一个完整的文本文件：

1. `.mergebeacon/ai-review.md`
2. `.mergebeacon/rules.md`
3. `AGENTS.md`
4. `CLAUDE.md`
5. `.github/copilot-instructions.md`
6. `.github/ai-review.md`

二进制文件、空文件、读取失败或被后端标记为截断的文件不会作为规则使用。规则文件内容会作为不可信
的评审上下文发送给 AI，不能改变系统安全约束，也不会直接触发远端操作。当前用户在页面编辑的本地
规则仍按平台、仓库隔离保存在本机；两类规则会在评审请求中分别标明来源。

评审历史和统一草稿目前只保存在本机 `localStorage`，按平台、仓库、PR/MR 和草稿的 `head_sha`
隔离。它们不会上传到 MergeBeacon 服务，也不会因为仓库规则文件的存在而自动进入 Git 仓库。

## 不在当前版本支持

- 团队成员之间共享 AI 历史、草稿或建议采纳状态。
- 跨设备同步本地规则、历史或草稿。
- 将完整 Diff、Token、AI API Key 或私有仓库正文上传到新的同步服务。
- 把同步来的旧建议直接当作当前版本可提交内容。

## 后续服务端方案

实现团队同步前需要单独确定身份、授权和数据边界。建议的最小模型如下：

- 以用户、团队、平台、仓库、PR/MR 和 `head_sha` 作为历史隔离键，并记录模型、评审焦点和 Diff
  是否截断。
- 默认同步评审摘要、建议和人工 action，不同步完整 Diff；私有仓库正文按团队策略加密并可关闭。
- 草稿同步采用版本号和明确的冲突状态，不自动覆盖本地未提交内容。
- 服务端提供保留期、删除、导出和团队管理员可见性控制，记录规则来源和规则版本。
- 使用 GitHub App、GitLab OAuth 或 Gitee 对应授权建立仓库访问边界，不复用平台 Token 到同步服务。
- 任何历史建议或同步草稿在提交前仍必须重新校验当前 `head_sha`、文件路径、行范围和平台权限。

在上述决策落地并有服务端 API 后，再增加同步存储和离线合并实现；在此之前仓库规则文件是唯一可审查、
可复现的团队共享入口。
